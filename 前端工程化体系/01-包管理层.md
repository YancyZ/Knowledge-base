# 01 · 包管理层（基础设施）

## Node.js：前端开发的运行环境

### 1.1 Node.js 是什么？

**Node.js** 是一个基于 Chrome V8 引擎的 JavaScript **运行时**，让 JavaScript 可以在**操作系统层面**运行，而不仅限于浏览器。

对前端开发者而言，Node.js 主要用于：

| 用途 | 举例 |
|------|------|
| 运行构建工具 | Vite、Webpack、Babel |
| 包管理 | npm、pnpm 安装依赖 |
| 本地开发服务器 | `vite dev` |
| 脚本自动化 | 代码检查、测试、部署脚本 |

> **重要**：Node.js 版本 ≠ 浏览器支持的 JS 版本。构建工具对 Node 版本有最低要求，例如 Vite 5 要求 Node 18+。

### 1.2 如何安装 Node.js？

**方式一：官网安装包**

访问 [nodejs.org](https://nodejs.org)，下载 LTS（长期支持）版本安装。

**方式二：使用 nvm（推荐）**

### 1.3 验证安装

```bash
node -v    # 例如 v20.11.0
npm -v     # 例如 10.2.4
```

## npm scripts 与生命周期

### 2.1 scripts 字段语义

`package.json` 的 `scripts` 是项目**可重复命令**的入口，CI 与开发者共用同一套脚本名：

```json
{
  "scripts": {
    "dev": "vite",
    "build": "tsc -b && vite build",
    "preview": "vite preview",
    "lint": "eslint src --max-warnings 0",
    "typecheck": "tsc --noEmit",
    "test": "vitest run",
    "prepare": "husky"
  }
}
```

| 钩子 | 触发时机 | 典型用途 |
|------|----------|----------|
| `preinstall` | install 前 | 环境检查（慎用） |
| `postinstall` | install 后 | patch、原生模块编译 |
| `prepare` | install / pack 前 | Husky 安装 |
| `prepublishOnly` | publish 前 | 构建、测试门禁 |

**注意**：`postinstall` 若执行网络请求或编译，会拖慢 CI — 能移入显式 `build` 脚本则移入。

### 2.2 脚本组合与跨平台

```json
"build:analyze": "cross-env ANALYZE=true vite build"
```

Windows 与 Unix 环境变量语法不同 — 使用 `cross-env` 保证一致。

### 2.3 传参

```bash
pnpm test -- --grep "UserProfile"
pnpm vite build -- --mode staging
```

`--` 之后参数传给底层命令。

## nvm：Node 版本管理器

### 3.1 为什么需要 nvm？

不同项目可能要求不同 Node 版本：

- 项目 A：Node 16（老项目）
- 项目 B：Node 20（新项目）

如果全局只装一个版本，切换项目时容易踩坑。**nvm**（Node Version Manager）让你在同一台机器上安装多个 Node 版本并快速切换。

### 3.2 常用命令

```bash
# 安装指定版本
nvm install 20
nvm install 18

# 切换版本
nvm use 20

# 查看已安装版本
nvm ls

# 设置默认版本
nvm alias default 20
```

### 3.3 Windows 用户

Windows 可使用 [nvm-windows](https://github.com/coreybutler/nvm-windows) 或 [fnm](https://github.com/Schniz/fnm)（跨平台，速度更快）。

### 3.4 项目级版本锁定

在 `package.json` 中声明引擎要求：

```json
{
  "engines": {
    "node": ">=18.0.0",
    "pnpm": ">=8.0.0"
  }
}
```

配合 `.nvmrc` 文件（内容为版本号，如 `20`），团队成员进入项目后执行 `nvm use` 即可对齐版本。

## 包管理器：npm、Yarn、pnpm

### 4.1 什么是包管理器？

前端项目会依赖大量第三方库（React、Vue、工具库等）。**包管理器**负责：

1. 从 npm  registry（默认仓库）下载包
2. 解析依赖树（A 依赖 B，B 依赖 C……）
3. 写入 `node_modules` 目录
4. 生成 lock 文件锁定精确版本

### 4.2 三者对比

| 特性 | npm | Yarn | pnpm |
|------|-----|------|------|
| 速度 | 中等 | 较快 | **最快** |
| 磁盘占用 | 每个项目完整复制 | 每个项目完整复制 | **硬链接共享存储** |
| 依赖隔离 | 扁平化，可能幽灵依赖 | 扁平化 | **严格，只能访问声明的依赖** |
| Monorepo | workspaces 支持 | workspaces 支持 | **原生 workspace，体验好** |
| 生态 | 默认随 Node 安装 | 需单独安装 | 需单独安装 |

**幽灵依赖**：项目未在 `package.json` 声明某包，却能 `import` 成功（因为被其他包间接装到了扁平的 `node_modules` 顶层）。pnpm 通过符号链接结构避免此问题。

### 4.3 常用命令对照

| 操作 | npm | Yarn | pnpm |
|------|-----|------|------|
| 初始化 | `npm init` | `yarn init` | `pnpm init` |
| 安装全部依赖 | `npm install` | `yarn` | `pnpm install` |
| 添加依赖 | `npm install react` | `yarn add react` | `pnpm add react` |
| 添加开发依赖 | `npm install -D vite` | `yarn add -D vite` | `pnpm add -D vite` |
| 移除依赖 | `npm uninstall react` | `yarn remove react` | `pnpm remove react` |
| 运行脚本 | `npm run dev` | `yarn dev` | `pnpm dev` |
| 全局安装 | `npm i -g pnpm` | `yarn global add` | `pnpm add -g` |

### 4.4 选型建议

- **新项目推荐 pnpm**：速度快、省磁盘、依赖严格
- 老项目若无迁移计划，保持原有包管理器，**不要混用**
- 团队统一一种，并在 README 中写明

## package.json 详解

`package.json` 是项目的「身份证」，描述项目元信息与依赖。

```json
{
  "name": "my-app",
  "version": "1.0.0",
  "private": true,
  "type": "module",
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview",
    "lint": "eslint src --ext .ts,.tsx"
  },
  "dependencies": {
    "react": "^18.3.1",
    "react-dom": "^18.3.1"
  },
  "devDependencies": {
    "typescript": "^5.4.0",
    "vite": "^5.2.0"
  },
  "engines": {
    "node": ">=18"
  },
  "packageManager": "pnpm@9.0.0"
}
```

### 5.1 关键字段

| 字段 | 含义 |
|------|------|
| `name` | 包名（发布到 npm 时须唯一） |
| `version` | 当前版本号（遵循 SemVer） |
| `private` | `true` 表示不发布到 npm |
| `type: "module"` | 使用 ES Module（`.js` 按 ESM 解析） |
| `scripts` | 可执行脚本，`npm run xxx` 调用 |
| `dependencies` | **生产依赖**（打包进产物） |
| `devDependencies` | **开发依赖**（构建、测试、Lint 等） |
| `peerDependencies` | 宿主环境应提供的依赖（如插件要求 React 版本） |
| `engines` | 要求的 Node / 包管理器版本 |
| `packageManager` | 锁定包管理器版本（Corepack 支持） |

### 5.2 dependencies vs devDependencies

```json
{
  "dependencies": {
    "axios": "^1.6.0"       // 运行时需要的库
  },
  "devDependencies": {
    "eslint": "^8.57.0",    // 仅开发时使用
    "vitest": "^1.4.0"
  }
}
```

**原则**：只在开发阶段用的放 `devDependencies`，否则会增加生产安装体积（虽然构建工具会做 tree-shaking，但语义上应区分）。

## Lock 文件：为什么必须提交？

### 6.1 问题场景

`package.json` 中写 `"react": "^18.3.1"`，表示允许安装 `18.3.x` 中任意满足 `^` 规则的版本。  
不同时间、不同机器 `install` 可能得到**不同的小版本**，导致「本地能跑、CI 却挂」。

### 6.2 Lock 文件的作用

| 包管理器 | Lock 文件 |
|----------|-----------|
| npm | `package-lock.json` |
| Yarn | `yarn.lock` |
| pnpm | `pnpm-lock.yaml` |

Lock 文件记录**每一层依赖的精确版本与下载地址**，保证所有人、CI 环境安装结果一致。

### 6.3 最佳实践

- ✅ **必须提交** lock 文件到 Git
- ❌ 禁止删除 lock 后随意推送
- 升级依赖后重新 install 并提交更新后的 lock
- CI 使用 `pnpm install ，frozen-lockfile`（或 npm/yarn 等价命令）确保与 lock 一致

## SemVer 语义化版本规范

### 7.1 版本号格式

```
主版本号.次版本号.修订号
MAJOR.MINOR.PATCH
例如：2.4.1
```

| 段 | 何时递增 | 含义 |
|----|----------|------|
| MAJOR | 不兼容的 API 变更 | 大版本，可能 breaking change |
| MINOR | 向下兼容的功能新增 | 新特性 |
| PATCH | 向下兼容的问题修复 | Bug 修复 |

### 7.2 版本前缀符号

在 `package.json` 中常见：

| 符号 | 示例 | 含义 |
|------|------|------|
| 精确 | `"1.2.3"` | 只装 1.2.3 |
| `^` | `"^1.2.3"` | 允许 `>=1.2.3 <2.0.0`（最常用） |
| `~` | `"~1.2.3"` | 允许 `>=1.2.3 <1.3.0`（更保守） |
| `*` / `latest` | `"*"` | 任意版本（**不推荐**） |

### 7.3 实战示例

```json
"react": "^18.3.1"
```

- 可安装 `18.3.1`、`18.4.0`、`18.9.9`
- 不会自动升到 `19.0.0`（主版本不同，可能不兼容）

### 7.4 预发布版本

- `1.0.0-alpha.1` — 内测
- `1.0.0-beta.2` — 公测
- `1.0.0-rc.1` — 候选发布

生产项目慎用 pre-release 依赖。

## 依赖安装原理（简化版）

```mermaid
flowchart LR
    A[package.json] --> B[包管理器解析]
    B --> C[查询 registry]
    C --> D[下载 tarball]
    D --> E[写入 node_modules]
    E --> F[生成/更新 lock 文件]
```

1. 读取 `package.json` 中的依赖声明
2. 递归解析每个包的 `dependencies`
3. 从 registry（默认 https://registry.npmjs.org）下载
4. 解压到 `node_modules`
5. 写入 lock 文件

### 8.1 node_modules 是什么？

存放所有已安装包的目录。**不应手动修改**，也不应提交到 Git（已在 `.gitignore` 中忽略）。

### 8.2 国内加速

可配置镜像源：

```bash
# pnpm
pnpm config set registry https://registry.npmmirror.com

# npm
npm config set registry https://registry.npmmirror.com
```

## 依赖解析与存储机制

### 10.1 npm / Yarn 的扁平化与幽灵依赖

npm v3+ 采用**扁平化** `node_modules`：尽量将依赖提升到顶层，减少嵌套深度。

```plaintext
node_modules/
├── react/              ← 直接依赖
├── lodash/             ← 被 react 的依赖提升上来
└── some-lib/
    └── node_modules/   ← 版本冲突时才嵌套
        └── react@17/   ← 与顶层 react@18 共存
```

**问题**：

1. **幽灵依赖（Phantom Dependency）**：业务代码 `import 'lodash'`，但 `package.json` 未声明 lodash，仍能解析成功（因被其他包装到顶层）
2. **重复安装**：同一包两个 major 版本共存，bundle 可能被打入两份
3. **不确定性**：不同包管理器、不同 install 顺序可能导致结构差异

**治理**：pnpm 严格模式；或在 npm 项目用 `eslint-plugin-import` 禁止未声明依赖；`depcheck` 定期扫描。

### 10.2 pnpm 内容寻址存储（CAS）

pnpm 全局 store（默认 `~/.pnpm-store`）以 **content-addressable** 方式存 tarball 解压内容，项目内 `node_modules` 通过**硬链接 / 符号链接**指向 store：

```plaintext
~/.pnpm-store/v3/files/00/xxxxx...   ← 物理文件（唯一）
project/node_modules/.pnpm/react@18.3.1/node_modules/react  ← 硬链接
project/node_modules/react → .pnpm/react@18.3.1/node_modules/react  ← symlink
```

**收益**：100 个项目依赖同版本 react，磁盘只存一份。  
**严格隔离**：业务只能访问 `package.json` 声明的依赖；未声明则 `import` 失败，这是 intentional 设计。

**.npmrc 关键配置**：

```ini
strict-peer-dependencies=true
registry=https://registry.npmmirror.com
```

### 10.3 依赖解析算法（简化模型）

包管理器解析可建模为 **SAT 约束求解**：读取声明 → 递归 dependencies / peerDependencies → 求满足所有约束的版本集合。

**peerDependencies 传递链**：

```plaintext
your-app → ui-lib@2 (peer: react@^18)
your-app → react@18.3.1  ✅

your-app → ui-lib@2 (peer: react@^18)
your-app → react@17     ⚠️ peer warning / CI 失败
```

React 18 起 npm 自动安装 peer；pnpm 需显式声明或配置 `auto-install-peers=true`。

### 10.4 overrides / resolutions（场景与写法）

当传递依赖存在 **CVE** 或同一库多版本共存导致构建异常：

```json
{
  "pnpm": {
    "overrides": {
      "lodash": "4.17.21",
      "webpack>acorn": "8.11.0",
      "react": "$react"
    }
  }
}
```

Yarn Berry 使用 `resolutions` 字段，语义类似。

**使用原则**：

- PR 中须注释：因 CVE-XXXX 锁定 / 因包 A 与包 B 版本冲突
- 定期尝试移除 overrides，验证上游是否已修复
- `"react": "$react"` 表示强制与子项目声明的 react 版本对齐

### 10.5 供应链安全（可操作清单）

**审计**：

```bash
pnpm audit
pnpm audit --audit-level=high
```

**依赖引入 Review 检查项**：

- 包名是否 typosquatting（如 `lodash` vs `lodahs`）
- 每周下载量、维护者数量、最近更新时间
- `postinstall` 脚本是否可疑（`preinstall`/`postinstall` 跑 curl）
- 源码体积是否异常小或 obfuscate

**工程门禁**：

- CI `pnpm install ，frozen-lockfile`
- Dependabot / Renovate 自动 PR
- 私有 npm proxy 缓存 + 扫描
- 禁止 CI Token 写权限过大

**典型事件响应**：CVE 披露 → overrides 锁安全版本 → 全仓 lock 更新 → 回归测试 → 热修发布 → 移除 overrides（若上游已修）

### 10.6 lock 文件冲突与合并

多人改依赖时 `pnpm-lock.yaml` 易冲突：

```bash
# 推荐：以一方为准后重新 install
git checkout --theirs pnpm-lock.yaml
pnpm install
# 或 merge 后
pnpm install
git add pnpm-lock.yaml
```

**禁止**：手工编辑 lock 中 integrity hash；不懂结构时一律 `pnpm install` 重生。

### 10.7 SemVer 边界情况

| 情况 | 说明 |
|------|------|
| `0.x.y` | `0.y` 视为不稳定 API，`^0.1.2` 可能升到 `0.2.0` |
| 预发布 | `1.0.0-beta.1` 默认不会被 `^1.0.0` 匹配 |
| `peerDependencies` | 不自动安装（pnpm 可配置 auto-install-peers） |

### 10.8 Node 版本管理器对比

| 工具 | 适用 |
|------|------|
| nvm / fnm | 个人多版本切换 |
| Volta | 团队按项目自动 pin 版本 |
| devcontainer | CI 与 onboarding 完全对齐 |

**Volta 示例**（进入目录自动切换）：

```json
{
  "volta": {
    "node": "20.11.0",
    "pnpm": "9.0.0"
  }
}
```

### 10.9 package.json exports 字段

现代包用 `exports` 限制公开 API，禁止 deep import 未导出路径：

```json
{
  "exports": {
    ".": "./dist/index.js",
    "./utils": "./dist/utils.js"
  }
}
```

### 10.10 Monorepo 依赖拓扑

```plaintext
apps/web → packages/ui → packages/utils
packages/ui 声明 peerDependencies: react（由 apps/web 提供）
```

反模式：`packages/ui` 把 react 放进 dependencies → 可能双份 React；环依赖 `utils → ui → utils`。

### 10.11 常见装包错误与处理

| 错误信息 | 原因 | 处理 |
|----------|------|------|
| `ERESOLVE unable to resolve` | peer 版本冲突 | 对齐 react 等 peer 版本或 overrides |
| `ENOENT ... lockfile` | 缺 lock | 提交 lock 或 CI 勿用 `--frozen-lockfile` 误配 |
| `Unsupported engine` | Node 版本过低 | nvm use / 升级 Node |
| `404 Not Found` | 私有包未配 registry | 检查 `.npmrc` |

### 10.12 诊断命令

```bash
pnpm why lodash
pnpm list --depth 3
pnpm dedupe --check
pnpm install --frozen-lockfile
```

## .npmrc 与 registry 配置

### 15.1 常用 .npmrc 项

```ini
registry=https://registry.npmmirror.com
strict-peer-dependencies=true
auto-install-peers=true
engine-strict=true
save-exact=false
```

| 配置 | 作用 |
|------|------|
| `registry` | 默认下载源 |
| `strict-peer-dependencies` | peer 冲突时 install 失败 |
| `engine-strict` | Node 版本不符则失败 |
| `@scope:registry` | 作用域包走私有源 |

### 15.2 私有 registry（企业）

```ini
@myorg:registry=https://npm.myorg.example.com/
//npm.myorg.example.com/:_authToken=${NPM_TOKEN}
```

CI 通过环境变量注入 Token，**禁止**写入仓库。

### 15.3 pnpm catalog（Monorepo 统一版本）

```yaml
# pnpm-workspace.yaml
catalog:
  react: ^18.3.1
  typescript: ^5.4.0
```

```json
"dependencies": {
  "react": "catalog:"
}
```

一处 bump，全仓对齐 — 减少版本漂移。

## optionalDependencies 与安装性能

### 16.1 optionalDependencies

平台相关原生模块（如 `fsevents` 仅 macOS）可声明为 optional — 安装失败不阻断整体 install。

### 16.2 CI 安装最佳实践

```bash
pnpm install --frozen-lockfile --prefer-offline
```

| 标志 | 含义 |
|------|------|
| `--frozen-lockfile` | lock 与 package.json 不一致则失败 |
| `--prefer-offline` | 优先本地缓存，加速 CI |
| `npm ci` | npm 等价，删 node_modules 重装 |

缓存策略：CI 缓存 `~/.pnpm-store` 或 `node_modules/.cache`，key 含 lock 文件 hash。

### 16.3 发布到 npm（库作者）

```bash
pnpm publish --access public --no-git-checks
```

发布前 checklist：README、LICENSE、`files` 字段白名单、`prepublishOnly` 跑 build+test、semver bump 与 changelog。

## 选型决策矩阵

| 场景 | 推荐 | 理由 |
|------|------|------|
| 新单体 SPA | pnpm + Corepack | 速度、严格、lock 稳定 |
| 10+ 包 Monorepo | pnpm workspace + Turborepo | 链接 + 缓存 |
| 遗留 npm 老项目 | 保持 npm，渐进迁移 | 混用 lock 风险 > 收益 |
| 安全敏感行业 | pnpm strict + audit CI 门禁 | 供应链可控 |

## 常见问题 FAQ

### Q1：`npm install` 很慢怎么办？

换镜像源，或改用 pnpm。

### Q2：能否在同一个项目里混用 npm 和 pnpm？

**不能**。混用会产生不同 lock 文件和 `node_modules` 结构，极易出问题。

### Q3：`dependencies` 里误加了开发工具怎么办？

移到 `devDependencies`：`pnpm add -D eslint` 并移除错误条目。

### Q4：如何查看过时的依赖？

```bash
pnpm outdated
```

### Q5：`node_modules` 删了会怎样？

重新 `pnpm install` 即可，lock 文件保证版本一致。

### Q6：什么是 peer dependency 警告？

表示某包期望宿主项目也安装特定版本的依赖（如 UI 组件库期望特定 React 版本）。按提示安装对应 peer 即可。

---

## 小结

包管理层是工程化链条的**第一道门禁**：Node 版本、包管理器、lock 文件三者对齐，CI 与本地才能装出同一棵依赖树。

提交 lock，CI 用 frozen install；理解 ^/~ 与 SemVer；pnpm 严格隔离避免幽灵依赖；peer 冲突用 overrides 并文档化原因；engines + .nvmrc 锁定运行时。

**易混点**：dependencies vs devDependencies；npm/Yarn 扁平化 vs pnpm 硬链接；混用包管理器；手工改 lock hash；0.x 版本的 ^ 规则。

核对：`pnpm why` 能否解释某传递依赖从哪来？audit 高危项是否有处理计划？私有包 .npmrc 是否未提交 Token？
