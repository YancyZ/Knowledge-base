# 06 · 性能优化与监控

## 性能为何属于工程化

### 1.1 业务影响

- **转化率**：延迟每增加 100ms，部分电商场景转化率可下降约 1%（行业统计量级，因场景而异）
- **SEO**：Core Web Vititals 为 Google 排名信号之一
- **留存**：卡顿与布局跳动直接损害信任感
- **成本**：未压缩图片、重复依赖、过大 bundle 浪费带宽与 CDN 费用

### 1.2 三类性能问题

| 类型 | 用户感知 | 典型根因 |
|------|----------|----------|
| 加载性能 | 白屏久、首屏慢 | JS/CSS 体积、TTFB、未压缩资源 |
| 运行时性能 | 滚动卡、输入迟滞 | 长任务、频繁重排、大列表全量渲染 |
| 资源效率 | 发热、低端机崩溃 | 内存泄漏、未释放监听器、大图 |

## Core Web Vitals 详解

Google 2024 起用 **INP** 替代 FID 作为交互指标。

### 2.1 LCP（Largest Contentful Paint）

**定义**：视口内**最大内容元素**（通常大图、大块文本、背景图 block）完成渲染的时间。

**候选元素**：`<img>`、`<video>` poster、`background-image` 块级元素、包含文本节点的块级元素。

**良好**：≤ 2.5s（p75）；需改进 2.5–4s；差 > 4s。

**常见拖累**：

- 服务器 TTFB 高
- 阻塞渲染的 CSS/JS
- 未优化 LCP 图片（过大、未 preload、lazy 误用在首屏）
- 客户端渲染等待 API 才渲染主内容

**优化清单**：

```html
<link rel="preload" as="image" href="/hero.webp" fetchpriority="high" />
<img src="/hero.webp" width="1200" height="600" alt="..." />
```

- 服务端/CDN 缓存 HTML
- 关键 CSS 内联或优先
- 首屏数据 SSR 或静态化

### 2.2 INP（Interaction to Next Paint）

**定义**：用户交互（点击、按键）到浏览器**呈现下一帧**的延迟；取页面生命周期内较差交互的分位值（通常 p75）。

**良好**：≤ 200ms；需改进 200–500ms；差 > 500ms。

**根因**：

- 主线程 > 50ms **长任务**（Long Task）
- 大量同步 JS、复杂 React 重渲染
- 强制同步 layout

**优化**：

- 拆分事件 handler 逻辑；debounce 输入搜索
- `startTransition` 标记低优先级更新（React）
- Web Worker 处理纯计算
- 虚拟列表

### 2.3 CLS（Cumulative Layout Shift）

**定义**：页面生命周期内**非预期布局偏移**累积分数。元素在没有用户交互时移动即计分。

**常见原因**：

- 图片/iframe 无 width/height
- Web 字体 FOIT/FOUT 导致文字 reflow
- 动态插入 banner、广告
- 骨架屏与真实内容高度不一致

**优化**：

```html
<img src="a.webp" width="800" height="450" alt="" />
```

```css
@font-face {
  font-family: 'MyFont';
  src: url('/font.woff2') format('woff2');
  font-display: swap;
  size-adjust: 100%; /* 可选，减少 swap 跳动 */
}
```

- 为异步内容预留占位高度
- 避免在现有内容上方插入 DOM（除非用户触发）

### 2.4 辅助指标

| 指标 | 含义 |
|------|------|
| **FCP** | 首次任意文本/图像绘制 |
| **TTFB** | 首字节时间（网络+服务端） |
| **TTI** | 主线程空闲可响应（Lighthouse 常用） |
| **TBT** | Total Blocking Time，长任务阻塞总和 |

### 2.5 实验室 vs 现场（RUM）

| | Lighthouse / WebPageTest | RUM（web-vitals） |
|---|--------------------------|-------------------|
| 环境 | 固定设备/网络模拟 | 真实用户设备、网络 |
| 用途 | CI、开发调试 | 生产监控、SLA |
| 局限 | 可能与用户不符 | 需采样与脱敏 |

**结论**：两者都要 — CI 防回归，RUM 看真实 p75/p95。

## 测量工具与实践

### 3.1 Chrome DevTools

**Performance 面板**：录制 → 查看 Main 线程火焰图、Long Task 红三角、Layout/Paint 事件。

**Network 面板**：瀑布图、Priority、是否 from cache、Timing 分解。

**Coverage 面板**：运行后查看 JS/CSS **未使用比例** — 指导 code splitting。

### 3.2 Lighthouse

```bash
npx lighthouse https://your-app.example.com --preset=desktop --view
npx lighthouse https://your-app.example.com --preset=perf --throttling-method=simulate
```

关注 Performance 分数下的 **Opportunities** 与 **Diagnostics**，而非只盯总分。

### 3.3 web-vitals 上报

```typescript
import { onCLS, onINP, onLCP, onFCP, onTTFB } from 'web-vitals';

function send(metric: Metric) {
  const body = JSON.stringify({
    name: metric.name,
    value: metric.value,
    rating: metric.rating,
    id: metric.id,
    page: location.pathname,
    deviceMemory: (navigator as any).deviceMemory,
    effectiveType: (navigator as any).connection?.effectiveType,
  });
  navigator.sendBeacon('/rum/vitals', body);
}

onLCP(send);
onINP(send);
onCLSS(send);
onFCP(send);
onTTFB(send);
```

## 加载性能优化

### 4.1 JavaScript 体积

**代码分割**：

```typescript
const Admin = lazy(() => import('./views/Admin'));
```

**manualChunks 示例**（Vite/Rollup）：

```typescript
rollupOptions: {
  output: {
    manualChunks(id) {
      if (id.includes('node_modules')) {
        if (id.includes('echarts')) return 'echarts';
        if (id.includes('react') || id.includes('react-dom')) return 'react-vendor';
        return 'vendor';
      }
    },
  },
},
```

**Tree-shaking**：ESM + `package.json` `"sideEffects": false`；避免 `import _ from 'lodash'` 整库。

**分析**：

```bash
pnpm build
npx vite-bundle-visualizer
# 或 rollup-plugin-visualizer
```

### 4.2 关键渲染路径（CRP）

最小化**阻塞资源**数量与体积：

1. 精简首屏 critical CSS
2. 非关键 JS 延迟：`defer` / 路由 lazy
3. 字体子集化 + `font-display: swap`
4. 避免首屏同步加载巨型 chart/map 库

### 4.3 资源提示

| 关系 | 作用 |
|------|------|
| `preload` | 当前页很快需要的资源 |
| `prefetch` | 下一导航可能用，低优先级 |
| `preconnect` | 提前建连（含 TLS） |
| `dns-prefetch` | 仅 DNS |

```html
<link rel="preconnect" href="https://api.example.com" crossorigin />
<link rel="modulepreload" href="/assets/vendor-abc123.js" />
```

### 4.4 图片与媒体

| 格式 | 场景 |
|------|------|
| AVIF / WebP | 照片 |
| SVG | 图标、简单矢量 |
| 响应式 srcset | 多 DPR/宽度 |

```html
<picture>
  <source srcset="/img/hero.avif" type="image/avif" />
  <source srcset="/img/hero.webp" type="image/webp" />
  <img src="/img/hero.jpg" width="1200" height="630" loading="lazy" alt="" />
</picture>
```

**注意**：LCP 图片**不要** lazy；应 preload + `fetchpriority="high"`。

### 4.5 压缩与传输

- 构建：Terser / esbuild minify
- 服务端：`gzip` 或 **Brotli**（通常比 gzip 小 15–20%）

```nginx
gzip on;
gzip_types text/plain text/css application/javascript application/json;
brotli on;
brotli_types text/plain text/css application/javascript application/json;
```

## 运行时性能优化

### 5.1 React

- **Profiler** 找重渲染组件
- **状态下沉**：避免父 state 变导致整树重渲染
- **Context 拆分**：高频更新勿放全局 Context
- **memo / useMemo / useCallback**：Profiler 证明有效再用
- **startTransition / useDeferredValue**：搜索、Tab 切换
- **虚拟列表**：`@tanstack/react-virtual`

### 5.2 Vue

- **v-memo**：依赖不变跳过子树更新
- **shallowRef / shallowReactive**：大对象浅响应
- **keep-alive** + 合理 `include`
- **虚拟滚动**：vue-virtual-scroller 等

### 5.3 长任务与 INP

```javascript
// PerformanceObserver 监控
const obs = new PerformanceObserver((list) => {
  for (const e of list.getEntries()) {
    if (e.duration > 50) console.warn('Long task', e.duration, e);
  }
});
obs.observe({ type: 'longtask', buffered: true });
```

**拆分策略**：

```javascript
function processChunk(items, i = 0) {
  const end = Math.min(i + 100, items.length);
  for (; i < end; i++) work(items[i]);
  if (i < items.length) requestIdleCallback(() => processChunk(items, i));
}
```

### 5.4 Web Worker

```typescript
// worker.ts
self.onmessage = (e) => {
  const result = heavyCompute(e.data);
  self.postMessage(result);
};

// main
const worker = new Worker(new URL('./worker.ts', import.meta.url), { type: 'module' });
worker.postMessage(payload);
worker.onmessage = (e) => setResult(e.data);
```

适合：大 CSV 解析、加解密、复杂几何计算 — 不适合频繁 DOM 操作。

### 5.5 内存泄漏常见模式

- 未 `removeEventListener`
- 未 `clearInterval` / `clearTimeout`
- 闭包持有已卸载组件引用
- 全局 Map 只增不减
- Observer / Subscription 未 unsubscribe

DevTools → Memory → Heap snapshot → 对比两次快照间 Detached DOM tree。

## 缓存与数据层

### 6.1 静态资源

- `app.[hash].js` → `Cache-Control: immutable, max-age=31536000`
- `index.html` → `no-cache`（每次验证新版本）

### 6.2 API 数据

- HTTP 缓存：`Cache-Control: max-age=60` 适合读多写少
- 客户端：TanStack Query `staleTime` / `gcTime` 减少重复请求
- SWR 模式：stale-while-revalidate

## SSR 与 Hydration 性能

### 10.1 水合成本

SSR 输出 HTML 后，客户端仍须下载 JS 并绑定事件。HTML 先到 ≠ 可交互。

| 策略 | 说明 |
|------|------|
| 流式 SSR | 分块 HTML，提前 FCP |
| Selective Hydration | 优先水合可见区域 |
| Islands | 仅交互岛 hydration |

### 10.2 避免 Hydration Mismatch

- SSR 禁止无种子 `Date.now()` / `Math.random()`  
- `window` / `localStorage` 仅客户端 `useEffect` / `onMounted`  
- 文本空白字符须服务端与客户端一致  

### 10.3 第三方脚本

统计、客服 widget 阻塞主线程 — 用 `async`、idle 加载或用户触发后再 insert。首屏不加载重型地图 SDK。

### 10.4 移动端与低端机

RUM 按 `deviceMemory`、`effectiveType` 分群；虚拟列表减小 overscan；`prefers-reduced-motion` 降级动画。

## LCP 专项排查

### 11.1 分段时间

| 子项 | 优化方向 |
|------|----------|
| TTFB | CDN、SSR、后端 |
| Load delay | preload、提高优先级 |
| Load time | 压缩、WebP/AVIF |

- LCP 元素使用 `loading="lazy"`  
- 首屏 hero 仅用 CSS background 无 `<img>`

1. Performance 找 >50ms Long Task  
2. 大 JSON 同步 parse → Worker 或分片  
3. React Profiler 查 commit 耗时  
4. 输入 debounce + `startTransition`

## FAQ

**Q：Lighthouse 90 分但用户投诉慢？**  
看 RUM p95、特定国家/机型；实验室 Throttling 可能与真实不符。

**Q：先优化 bundle 还是先优化图片？**  
用 Lighthouse/Analyzer 看 LCP 元素 — 若 LCP 是图，优先图；若主线程 long task 多，优先 JS。

**Q：SSR 一定更快？**  
SSR 改善首屏 HTML，但 hydration 增加 JS 成本；需对比 TTFB + LCP + TTI。

## 小结

性能优化要**先度量再动手**：Web Vitals 定义体验目标，构建与运行时优化围绕 LCP/INP/CLS 展开，监控闭环验证改动有效。

路由级 code splitting；图片 WebP/AVIF + 尺寸；长列表虚拟化；关键 CSS/字体 preload；RUM 上报 Core Web Vitals。

**易混点**：只看 Lighthouse 单次实验；优化未上线指标；第三方脚本阻塞主线程；CLS 来自无尺寸图片/字体。

核对：生产 RUM 与 lab 数据是否对齐？慢接口是否有 Skeleton 而非白屏？
