# 数据的表示

计算机只认**比特**；整数、浮点、字符在内存里都是位模式。搞清进制、**补码**与 **IEEE 754**，才能解释 `0.1 + 0.2 !== 0.3`、位运算、TypedArray 与二进制协议。

---

## 进制与位

日常十进制、调试十六进制、硬件二进制，本质是同一数值的不同写法。前端里颜色值、权限掩码、协议 flag 都直接操作位。

| 进制 | 用途 |
|------|------|
| **二进制** | 硬件、位标志、掩码 |
| **十六进制** | 紧凑表示字节（颜色 `#FF00AA`、MAC 地址） |
| **十进制** | 人类读写、业务展示 |

```plaintext
一个字节 = 8 bit = 2 个十六进制 digit
0xFF = 1111 1111 = 255
0xA5 = 1010 0101 = 165
```

**位运算**（JS 按 32 位有符号整数解释 `|` `&` `^` `~` `<<` `>>` `>>>`）：

```javascript
0b1010 & 0b1100;   // 0b1000 = 8  按位与，取交集位
1 << 3;            // 8         左移 = ×2^n
( -1 >>> 0 );      // 4294967295 无符号右移，高位填 0
0xFF00 >> 8;       // 0xFF      取高字节
```

权限位、颜色通道、协议 flag 常用掩码：`value & MASK` 取字段，`value | FLAG` 置位，`value & ~FLAG` 清位。

## 有符号整数与补码

n 位补码表示范围：**[-2^(n-1), 2^(n-1)-1]**。现代 CPU 用补码统一加减，加法器不必单独做减法电路。

| 概念 | 说明 |
|------|------|
| **原码** | 符号位 + 绝对值（教材对比用，硬件不用） |
| **反码** | 符号位不变，其余取反 |
| **补码** | 反码 + 1；0 的表示唯一 |

```plaintext
8 位: +1 = 0000 0001
      -1 = 1111 1111  (补码)
+1 + (-1) 丢弃进位 → 0000 0000
```

**溢出**：超过表示范围会**回绕**（wrap around），无符号与有符号规则不同。

```javascript
const u8 = new Uint8Array(1);
u8[0] = 255;
u8[0]++;           // 0  无符号 8 位溢出
(127 + 1) | 0;     // -128  有符号 32 位视角
```

| 类型 | 字节 | 约范围 |
|------|------|--------|
| `int8` / `uint8` | 1 | ±128 / 0–255 |
| `int16` / `uint16` | 2 | ±32K / 0–64K |
| `int32` / `uint32` | 4 | ±2G / 0–4G |
| `int64` | 8 | 很大（JS 用 `BigInt`） |

Web 里 **ArrayBuffer / TypedArray / DataView** 按指定宽度读写；Wasm、图片像素、音频采样都依赖精确布局。超过 `Number.MAX_SAFE_INTEGER`（2⁵³-1）的 id 应用 string 或 `BigInt`。

## 浮点数：IEEE 754

常见 **64 位 double**：1 符号 + 11 指数 + 52 尾数（隐含 leading 1）。32 位 float 布局类似，位数更少。

```plaintext
值 ≈ (-1)^s × 1.m × 2^(e - bias)
```

| 情况 | 指数 | 尾数 |
|------|------|------|
| 正规数 | 非全 0/1 | 隐含 1.xxx |
| **0** | 全 0 | 0 |
| **∞** | 全 1 | 0 |
| **NaN** | 全 1 | 非 0 |

**精度问题**：十进制 0.1 在二进制中是无限循环小数，无法精确存储。这不是 JS 的 bug，而是 IEEE 754 的通用行为。

```javascript
0.1 + 0.2;                    // 0.30000000000000004
Number.EPSILON;               // 浮点比较容差参考
Object.is(NaN, NaN);          // true
Object.is(+0, -0);            // false  符号位不同
```

| 场景 | 建议 |
|------|------|
| 金额 | 整数分存储，或 decimal 库 |
| 坐标/动画 | double 通常够用 |
| 比较相等 | `Math.abs(a-b) < eps` 或整数化 |
| GPU/WebGL | `Float32Array`，精度低于 double |

## 字符与编码

文本在内存中是**码点 → 编码字节**的映射。Web 传输与存储默认 UTF-8；JS 字符串内部是 UTF-16 码元序列。

| 编码 | 说明 |
|------|------|
| **ASCII** | 7 位，英文与控制符 |
| **UTF-8** | 变长 1–4 字节，Web 默认 |
| **UTF-16** | JS 字符串内部表示；BMP 外字符占 2 码元 |

```javascript
"𠮷".length;              // 2  辅助平面字符两个 UTF-16 码元
[..."𠮷"].length;          // 1  按 Unicode 码点
new TextEncoder().encode("中文").length;  // 6  UTF-8 字节数
```

按字素切分用 `[...str]`、`Intl.Segmenter` 或 `Array.from`；按字节处理用 `TextEncoder`/`TextDecoder`。

## 字节序（Endian）

多字节整数在内存中的存放顺序分**大端**（高位字节在低地址）与**小端**（低位字节在低地址）。x86、ARM 常见小端；网络协议常规定大端（网络字节序）。

```javascript
const buf = new ArrayBuffer(4);
const view = new DataView(buf);
view.setUint32(0, 0x12345678, false);  // 大端
view.setUint32(0, 0x12345678, true);   // 小端
// 读二进制协议时 DataView 的 littleEndian 参数必须对齐规范
```

## 定点数与 Q 格式

音频、嵌入式常用**定点**：小数点位置固定，用整数存 scaled 值，避免浮点单元。

```plaintext
Q15: 1 符号位 + 15 小数位，范围约 [-1, 1)
值 = int16 / 32768
```

Web Audio 内部部分 DSP 用 float32；IoT 协议字段可能是 int16 定点角度或温度，读 TypedArray 时要查协议文档的 scale 因子。

## 校验与冗余位

二进制协议常带**校验**检测传输错误：

| 方式 | 宽度 | 特点 |
|------|------|------|
| 奇偶校验 | 1 bit | 检单 bit 错 |
| CRC | 16/32 bit | 检突发错误 |
| 哈希 | 任意 | 完整性 + 防篡改 |

```javascript
// 简单 XOR 校验（教学用，非密码学安全）
function xorChecksum(bytes) {
  return bytes.reduce((a, b) => a ^ b, 0);
}
```

密码学哈希（SHA-256）与 HMAC 用于签名与完整性，与简单 XOR/CRC 用途不同。

## JSON 与 Number 精度边界

JSON 规范只有 number 类型，且常见解析器映射到 IEEE 754 double：

```javascript
JSON.parse('{"id":9007199254740993}').id;  // 9007199254740992  丢精度
JSON.stringify({ id: 9007199254740993n }); // TypeError  BigInt 默认不可序列化
```

| 策略 | 适用 |
|------|------|
| id 用 string | 雪花 ID、UUID |
| BigInt + 自定义 reviver | 内部工具链 |
| protobuf / MessagePack | 显式 int64 类型 |

## 与前端/Node 的衔接

| 场景 | 数据表示要点 |
|------|--------------|
| 颜色 / 权限 / protobuf | 位字段与掩码 |
| Canvas ImageData | `Uint8ClampedArray` 每通道 0–255 |
| 文件 / 网络二进制 | `Blob`、`ArrayBuffer`，注意 endian |
| JSON 数字 | 只有 double，超大整数丢精度 |
| Wasm 线性内存 | 按偏移读写 TypedArray |

```javascript
// 从网络读 uint32 BE
const view = new DataView(buffer);
const magic = view.getUint32(0, false);
```

## 小结

数据在底层都是比特；整数用**补码**统一加减，溢出会回绕；浮点按 **IEEE 754** 存储，十进制小数常不精确；文本是码点的编码字节序列。

**易混点**：JS `number` 全是 double，不是 decimal；`>>` 与 `>>>` 对负数不同；`NaN !== NaN` 但 `Object.is(NaN, NaN)` 为 true；UTF-8 字节数 ≠ 字符串 `.length`；CRC 校验 ≠ 密码学哈希。

核对：`Uint8` 255+1 得多少？为何 0.1+0.2 不等于 0.3？`int32` 与 `float32` 各几字节？JSON 为何不适合传 int64 主键？
