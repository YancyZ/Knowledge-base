## 例题：React 渲染流程

**定义**：触发更新后经历 render（可中断）与 commit（改 DOM）。

**机制**：
1. setState / 父 render 等调度更新  
2. render 阶段执行组件、Diff  
3. commit 改 DOM、跑 layout effect  
4. 浏览器绘制后 passive effect  

**例子**：`setCount` → 函数重跑 → 文本节点更新。

**权衡**：Concurrent 要可中断，避免长任务挡输入。

---

## 例题：useEffect 与 useLayoutEffect

| | useEffect | useLayoutEffect |
|---|-----------|-----------------|
| 时机 | paint 后 | paint 前 |
| 阻塞绘制 | 否 | 是 |
| 场景 | 拉数据、订阅 | 测 DOM、同步改 UI |

**别滥用 layout**，默认 useEffect。

---

## 例题：如何优化 React 性能

**分层答**：

1. **加载**：路由 lazy、减包  
2. **渲染次数**：状态下沉、memo、Context 拆分  
3. **渲染重量**：虚拟列表  
4. **交互**：startTransition  
5. **测量**：Profiler、web-vitals

## 例题：服务端状态放哪

**定义**：API 数据是服务端真相的缓存副本。

**答**：TanStack Query / SWR；不放 Redux 除非已有投资。

**对比**：客户端 UI 状态 → Zustand / useState。

---

## 例题：key 的作用

**答**：帮助 Diff 识别**同一层级**节点身份，重排时复用 DOM。

**反例**：index key 在重排、插入时导致状态错乱。

---

## 开放题：为什么 React 用函数组件

| 点 | 说明 |
|----|------|
| Hooks 组合逻辑 | 替代 HOC / Render Props |
| 无 this | 函数组件无实例 |
| 利于 Compiler 分析 | 纯函数更易优化 |
| 与 Concurrent 契合 | 可中断渲染友好 |

类组件仅维护 / Error Boundary。

## 小结

面试用「定义→机制→例子→权衡」结构；结合场景答，显 senior 判断。

答题框架：定义（30 秒共识）→ 机制（深度）→ 例子（代码）→ 权衡（工程判断）。高频题：渲染流程（render+commit）、useEffect vs useLayoutEffect、性能优化分层（加载/次数/重量/交互/测量）、服务端状态用 Query、key 帮助 Diff 识别身份。开放题：函数组件 + Hooks 组合逻辑、无 this、Compiler 友好、Concurrent 契合。手写准备：简易 useState/useEffect、防抖 Hook、复合组件。结构清晰 + 带权衡 > 背定义。
