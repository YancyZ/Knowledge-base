# 服务端状态本质

> 从 API 拉回来的数据，和「主题开关」「弹窗开闭」不是同一类 state。**服务端状态**有缓存、过期、去重、后台刷新等专属问题——用 `useEffect + useState` 手写往往又乱又慢。

---

## 一、客户端 state vs 服务端 state

```mermaid
flowchart LR
  subgraph client [客户端 state]
    UI[弹窗 / Tab / 主题]
    Auth[登录态 token 等]
  end
  subgraph server [服务端 state]
    List[用户列表]
    Detail[订单详情]
    Page[分页 / 筛选结果]
  end
  API[(REST / GraphQL)]
  server --> API
  API --> server
```

| 维度 | 客户端 state | 服务端 state |
|------|--------------|--------------|
| **来源** | 用户交互、本地配置 | 远程 API |
| **真相在哪** | 浏览器内存 | **服务器**（前端只是副本） |
| **是否共享** | 常需全局 store | 多组件可读同一份 cache |
| **过期** | 一般不过期 | 会 stale，需 refetch |
| **典型工具** | useState、Zustand | **TanStack Query**、SWR |

## 二、手写 fetch 的痛点

```tsx
function UserList() {
  const [users, setUsers] = useState<User[]>([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    let cancelled = false;
    setLoading(true);
    fetchUsers()
      .then(data => { if (!cancelled) setUsers(data); })
      .catch(e => { if (!cancelled) setError(e); })
      .finally(() => { if (!cancelled) setLoading(false); });
    return () => { cancelled = true; };
  }, []);

  if (loading) return <Spinner />;
  if (error) return <ErrorView error={error} />;
  return <ul>...</ul>;
}
```

| 问题 | 说明 |
|------|------|
| **重复请求** | 两个组件各写一遍 effect → 打两次 API |
| **无缓存** | 切路由回来又要 loading |
| **竞态** | 慢请求后返回覆盖新数据（需 cancelled 标志） |
| **样板代码** | loading / error / empty 每页重复 |
| **更新同步** | 改完用户资料，列表页还是旧数据 |

## 三、服务端 state 要解决什么？

```mermaid
flowchart TB
  Q[Query 库职责]
  Q --> C[缓存 Cache]
  Q --> D[去重 Dedupe]
  Q --> S[Stale 标记]
  Q --> R[后台 Refetch]
  Q --> I[失效 Invalidation]
  Q --> M[Mutation 后更新]
```

| 能力 | 用户感知 |
|------|----------|
| **缓存** | 二次进入页面「秒开」旧数据 |
| **去重** | 10 个组件同时要 users → 只请求 1 次 |
| **staleTime** | 5 分钟内认为数据新鲜，不重复打 |
| **refetchOnFocus** | 切回标签页自动更新 |
| **invalidate** | 提交表单后列表自动刷新 |

## 四、不要放进 Redux / Zustand 的数据

| ❌ 常见误用 | ✅ 推荐 |
|-------------|---------|
| `dispatch(fetchUsers())` 进 Redux | `useQuery(['users'], fetchUsers)` |
| Zustand 存整个商品列表 | Query cache |
| useState 复制 Query 的 `data` | 直接用 `data` |

**例外**：离线优先、复杂本地同步（如 Notion 类）可能自建 sync 层；一般 B 端后台不必。

## 五、与 URL state 的关系

URL 描述「要查什么」，Query 负责「查回来并缓存」：

```tsx
const [params] = useSearchParams();
const page = Number(params.get('page') ?? 1);

const { data } = useQuery({
  queryKey: ['orders', { page }],
  queryFn: () => fetchOrders({ page }),
});
```

见 [08-状态管理 · URL 状态](../08-状态管理/06-URL状态与路由参数.md)。

## 七、心智模型

> **服务端 state = 远程数据的本地 cache + 同步策略**，不是「又一个 global store」。

| 问自己 | 答案指向 |
|--------|----------|
| 数据来自 API？ | Query |
| 只本组件、不共享？ | 仍可用 Query（去重）或简单 fetch |
| 改 API 后要更新多处 UI？ | `invalidateQueries` / `setQueryData` |
